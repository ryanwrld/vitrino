# Itens Adiados — Fase 3

Itens descobertos durante a execução que estão fora do escopo da task/plan atual (não causados pelas mudanças desta execução), portanto não corrigidos automaticamente.

**Atualização 2026-07-14 — rate limit de signup, mitigado mas não 100% eliminado:** A suíte de testes foi migrada para um projeto Supabase DEDICADO e isolado (`TEST_SUPABASE_URL`/`TEST_SUPABASE_ANON_KEY`/`TEST_SUPABASE_SERVICE_ROLE_KEY` em `.env.local`, nunca o projeto de produção). `tests/setup/supabase-test.ts` cria a conta de teste via `admin.createUser` (service_role) quando a chave está configurada — bypassa o endpoint público de signup do GoTrue — mantendo a autenticação real via `signInWithPassword` pra RLS continuar sendo exercitada de verdade. Um `retryOnRateLimit` com backoff foi adicionado dentro de `seedAuthenticatedAccount` para absorver o teto de auth do tier Free (que não é totalmente controlável pelo campo Authentication > Rate Limits do Dashboard, mesmo subindo pra 500/5min — parece ser um limite de plataforma, não só o valor configurado).

Resultado: full suite foi de ~74/130 pra **91/130 passando**. Os ~38 restantes ainda falham por rate limit especificamente nos **13 arquivos que chamam `signUpAction`/`signInAction` diretamente** (`tests/products/*`, `tests/settings/*`, alguns `tests/auth/*`/`tests/onboarding/*`) — esses testam o Server Action real de produção ponta-a-ponta (não passam por `seedAuthenticatedAccount`), então o retry-with-backoff não os cobre. Estender o retry pra esses 13 arquivos é possível mas não foi feito (custo/benefício não justificado pro estágio atual de MVP sem billing — decisão consciente do usuário em 2026-07-14, não um "não corrigido" por falta de solução). Rodar via `npx vitest run --no-file-parallelism` reduz um pouco a variância mas não resolve sozinho.

## Plan 03-06

- **Mesmo bloqueador de rate limit do Supabase Auth (herdado do Plan 03-04), agora bloqueando a suíte completa no checkpoint final da fase.** Ao rodar `npm test` (a suíte inteira, 108 testes/25 arquivos) para o checkpoint da Task 4, praticamente TODO teste que faz `signUp` real falha com `"Request rate limit reached"` — não só `tests/products/`, mas também `tests/auth/`, `tests/onboarding/`, `tests/settings/`, `tests/rls/`. Isso confirma que é puramente um esgotamento de cota do projeto Supabase remoto de teste (sem relação com nenhuma mudança desta wave ou de waves anteriores): `tests/products/list-filter-sort.test.ts` (o arquivo desta plan) passa integralmente (1/1) quando rodado isolado, tanto logo após a Task 1/2/3 quanto em reruns isolados posteriores — só falha quando entra na fila da suíte completa, depois de dezenas de outros `signUp` já terem consumido a cota. Tentativas de mitigação nesta sessão: (1) aguardar o rate limit resetar (observado reset rápido, ~1-2min, para UM arquivo isolado de poucos signUps) e re-rodar; (2) rodar `npm test -- --no-file-parallelism` para reduzir concorrência de signUps simultâneos — nenhuma das duas foi suficiente para a suíte inteira (108 testes, dezenas de `signUp`) ficar verde de uma vez, porque o volume total de signups da suíte completa excede a cota de burst do projeto de teste independente de paralelismo. **Não corrigido — fora do escopo deste plano** (seria uma mudança de infraestrutura de teste, ex.: emulador local de Supabase Auth ou service_role para seed direto de usuários, ambas mudanças arquiteturais que exigem decisão do usuário, não um "auto-fix" de Rule 1-3). Reforça — pela terceira vez nesta fase — a necessidade de um emulador local de Supabase Auth (ou seed via `service_role`, contornando o GoTrue signup) para a suíte de testes antes que este projeto cresça mais.
