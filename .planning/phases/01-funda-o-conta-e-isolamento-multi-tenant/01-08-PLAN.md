<objective>
Corrigir o gap blocker (UAT teste 6): `createClient()` em `src/lib/supabase/server.ts` — a factory usada em Server Components, Server Actions e Route Handlers — implementa `setAll` sem o try/catch documentado pela orientacao oficial do `@supabase/ssr` para o App Router do Next.js. Quando `getUser()` e chamado a partir de um Server Component (ex: via `requireCompletedOnboarding()` em `onboarding-guard.ts`, invocado por `page.tsx`) e o token de sessao precisa de refresh, o cliente Supabase tenta escrever os cookies renovados via `setAll` -> `cookieStore.set()`. O App Router so permite escrita de cookies dentro de Server Actions e Route Handlers — nunca durante o render de um Server Component — entao a chamada lanca `Cookies can only be modified in a Server Action or Route Handler` (visto no Console ao recarregar /onboarding, apontando para server.ts:23).

Purpose: Recarregar qualquer pagina admin deve ser silencioso mesmo quando o token vence durante o render. A supressao e segura por design: `src/middleware.ts` (via `updateSession`) ja renova a sessao a cada request, entao o refresh redundante disparado por um Server Component pode ser ignorado sem nenhuma regressao de frescor de sessao.

Output: `setAll` envolve o `cookiesToSet.forEach(...)` em um try/catch (catch com comentario explicativo, sem re-throw), seguindo o padrao oficial documentado do `@supabase/ssr`. Teste unitario deterministico provando os dois caminhos: (a) contexto de Server Component (`.set()` lanca) NAO propaga a excecao; (b) contexto de Server Action/Route Handler (`.set()` sucede) ainda escreve todos os cookies corretamente.
</objective>
